"Featured Story"

User Freedom

"Why is there only one featured story (of the day)?"

We changed the starting view from one featured story to a "story feed". Clicking on ‘Read More’ opens up the profile in-place. The "Read More" button is then replaced with a "Start a Conversation" button.

Space Utilization

"Too much space on the local/featured stories page."

We decided to implement a version of the "Local Stories" view which showed information in an even more condensed form contained in clear "cards". Clicking on the cards leads to the user’s detailed profile (which also contains their stories)

User Connection

How is the featured story going to help me connect with people?

We put the "call to action" (start a conversation) as a big button which follows the user as they browse a user's stories and profile.

Confusing UI

Clicking on the chevron expands the user’s profile presented in a friendly manner which is easy to interpret. Scrolling down through the user’s story chapters keeps the "Start a Conversation with Audrey" icon stuck to the top - this is a very clear affordance and call to action to the user, funnelling them through to the one-to-one conversation phase.

"My Profile"

Profile Layout

"The profile photo is too big"

"Other users’ profiles should show the same information that is being asked for from the user in the welcome screen"

As a response, profile photos were reduced in size and important information is better presented. We have made several attempts in the grid division.

Edit Button

"The Edit button is unclear - does it mean edit profile? Edit chapters?"

This was fixed by adding a clearly distinguishable Edit button on each section - to clearly explain visually to the user that each section is editable separately.

Bottom Navigation Bar

This was the first set of icons that we designed for the bottom navigation bar (for the marvel mockup). In order, the icons represent "Featured stories", "My inbox/conversations", "Write a new chapter", "My profile".

"Icons should be more obvious, and definitely need text labels along with them."

Additionally, we felt that grouping the icons more logically would result in a more intuitive experience. We thus grouped "conversations" and "profiles" together. We also decided to keep the "featured stories" functionality separate, acting as a call to action for the user to explore it. The icons now have text explanations and are also highlighted in blue when a user is in the view represented by them.

"It is hard to understand when an icon was highlighted, and it is hard to read the text."

"Icons should be of the same shape, size and form and come from the same family. There is no reason for featured story to be any bigger than the other icons."

The icons are now of the same size, and have been custom modified to make them part of the same ‘family’.

"Icons are too busy, they are hard to understand"

"Why is the icon for conversations a cloud?"

We iterated and built a new set of custom icons. The final icons are legible, have no fine detail and clearly come from the same family. Changes in the nav bar also went hand-in-hand with a title bar on the top of the screen, which also shows the user which section of the app they are presently in.

"Sign-up/Welcome"

The signup screen was added in a later iteration of the prototype as a way to introduce the user to the application and ask them for their personal information. These are the very first mockups that we had - these were shown to expert reviewers and to users to get feedback.

The breadcrumb indicators at the bottom (Step 1, Step 2…) looked like buttons. They are only supposed to be breadcrumbs which indicate where in the signup flow the user is presently located. Not just how they look, but also the fact that they are positioned at the bottom makes it seem to a user that they are clickable and part of the navigation experience.

There needs to be a ‘previous’ button so that one can go back and make changes. The visual elements need to be consistent with one design language.

What to input?

"Adding location - can it be entered manually as well?"

"How is location information being taken?"

To clarify this to users, we added a small disclaimer that told them that their location was automatically taken, and also gave them the option to change it manually.

"Can you relate users at same stages of cancer (not by tags)?"

"Number of children should be binary - asking for varying number of children is annoying.

We ultimately removed the whole concept of tags altogether. As for children, we concluded that the dividing users up into broad binary classes of “no children” and “have children” is good enough to be useful to users - more granular levels of detail could be added by them to their bio if they are comfortable sharing it.

How to input?

"Ask users useful questions and mine that information to suggest other users to connect with."

We asked the user more useful questions about themselves in the signup screen and our system would mine this information to make suggestions.

Order of Input?

"Instead of asking the users to write their first chapter at the end of the welcome/signup process, ask them to write their Bio here. This will put them in the right frame of mind to write a bio (at the end of the signup process)."

We finalized upon the following design for the welcome screen. The design language was kept consistent with the rest of the app, and also with the 'edit profile' view in the 'my story' section.

"Tags"

Tags were introduced as a way for the user to “tag” chapters in their story into 6 different preset categories. This would enable other users to quickly understand the subject matter of each other's’ chapters and stories by looking at the tags which were used. The way a user would interact with tags is in the second step of the two-step process of creating a new chapter in one’s story.

"Why are tags so huge!"

"Six icons are not so easy to remember … they aren’t that memorable”

"Can tags be integrated into the chapter/story writing view on the same page?"

"Tags should be on the same page as the edit chapter."

"It’s not clear when a tag is selected or not.

When it was explained to the users that the purpose of tagging is to match and recommend users based on the themes of their stories encapsulated by tag counts, we got much more useful and meaningful feedback:

"What are you [also] collecting user information for? Can it be used with the tags to suggest other users?"

These suggestions went hand-in-hand with our upgradation of the user profiles view and the information we were collecting from users. Since our purpose with the tags was to enable users to connect with one another, we decided to do this a different way - through personalized recommendations from our application itself. In our user research, we had learned that breast cancer patients. We thus re-designed the local stories view to include personalized recommendations generated by our app. We received much more positive feedback after this change and users said it was much clearer what they need to do at this stage (in the Local Stories view).

Help/Guide

Explain Everything

"Should have explanation for everything"

We placed one or two sentences explaining the view at the top of each view. We took time to come up with a clear and friendly paragraph to guide users through signup view.

Distinguish "My story" and "Local Stories"

The appearances of local stories and my stories are very alike, therefore causing a lot of confusions.

Visual Design

Mixed use of line styles

"Dashed and dotted lines are not legible."

We notice an inconsistent use of line styles. We removed the dashed line between icons on the nav bar, and the dotted line inside the buttons for cleanliness. We only use dashed line for division between chapters, and everything else are solid lines

Tension between legibility and aesthetics

“The color looks not medical.”

“I’m not convinced that the font is legible enough.”

We decide on taking a stand in our font and color. We choose “Indie Flower” because it looks flowery, and works very well on the title of the views. “Coming Soon” is a handwritten font that conveys the feeling of “writing a story”. They are also quite readable. As to color choice, we think it good to keep ourselves away from “medical” app, because we want to encourage patients to share more than logistical experience. We hope our users to treat this app more like a “diary”.

Alignment and Grid

"left align profile details and use the grid."

We used grid and redesigned the layout for the profile "card". We realized that information is more well communicated to the user if "matched characteristics" are shown in the first place.
